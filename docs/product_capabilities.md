# 产品能力矩阵

本文档从商业化产品角度描述 Mahjong Ops Workflow 的能力。

## 产品定位

Mahjong Ops Workflow 是面向麻将馆和棋牌室的 AI 运营中台。它不是一个简单聊天机器人，也不是完全自治 Agent，而是一个围绕“组局成功率”和“客户运营效率”设计的可控 agentic workflow。

它的边界是：确定性后端负责状态、幂等、客户锁、outbox 和审计；AI 负责语义理解、弱表达归一、上下文依赖判断和话术辅助。

它承担的角色接近一个经验丰富的麻将馆店长：

- 看懂客户在群聊和私聊里的真实意图。
- 记住每个客户爱打什么、什么时候常来、一天能打几场。
- 判断当前缺人的局能不能拼。
- 知道该问谁、不该问谁、谁今天已经打过。
- 生成群发和私聊邀约草稿。
- 在不确定和高风险场景下及时转人工。

## 核心用户

- 单店麻将馆老板。
- 连锁棋牌室运营负责人。
- 私域微信群运营人员。
- 有多个群和客户池的麻将馆店员。
- 希望把组局、拉人、客户维护半自动化的门店。

## 核心场景

### 1. 自动识别组局需求

客户可能不会标准表达“我要组局”，而是说：

- `今天下班有人打麻将吗`
- `cq371 0.5 19.30 无烟`
- `川麻216三等一`
- `今晚有搭子没`
- 发语音、图片、表情包、截图

系统会把这些消息转换为结构化意图：

- 是否麻将相关。
- 是否找人。
- 当前已有几人。
- 缺几人。
- 什么玩法。
- 什么档位。
- 什么时间。
- 是否需要追问。

### 2. 待组局队列

当信息明确但还不完整时，系统不会乱邀约。

例如：

```text
川麻216三等一
```

系统识别为：

- 川麻
- 2-16 档
- 底注 2
- 封顶 16
- 三缺一
- 缺开局时间

动作：

- 进入待组局队列。
- 回复追问开局时间。
- 暂不生成私聊邀约。

### 3. 本地玩法知识

系统支持把门店本地行话配置为结构化规则。

当前已内置：

- `cq` = 杭麻财敲
- `371` = 三缺一
- `272` = 二缺二
- `173` = 一缺三
- `216` = 2-16 档，底注 2，封顶 16
- `132` = 1-32 档，底注 1，封顶 32

支持玩法：

- 杭麻
- 财敲
- 川麻
- 换三张
- 定缺
- 幺鸡
- 素鸡
- 幺鸡47
- 红中麻将
- 捉鸡麻将
- 湖南麻将
- 重庆麻将

这些不是写死在产品叙事里的固定上限，而是后续可以扩展为门店玩法词典。

### 4. 房态冲突处理

组局必须受房态约束。系统在生成邀约前会检查目标时间是否有房。

例如 16:00 时客户说：

```text
5点 0.5 三缺一 无烟
```

如果 17:00 当前满房，最快 18:00 有房，系统不会继续按 17:00 发群或私聊拉人，而是回复：

```text
17:00 这个时间目前满房，最快 18:00 有房。你看能不能改到 18:00 开？可以的话我按 18:00 继续帮你组；如果还是想 17:00，我先给你记候补，有房马上同步。
```

这种策略避免“人组好了但没有房间”的运营事故。

### 5. 客户画像

系统维护客户画像，用于判断“某个局缺人时优先问谁”。

画像包括：

- 常打玩法。
- 常打档位。
- 细分玩法偏好。
- 无烟/有烟偏好。
- 常见同行人数和置信度。
- 常打时间。
- 常打星期。
- 拒绝频率。
- 最近邀约时间。
- 每日可打场次。
- 两场之间建议间隔。
- 邀约冷却时间。

示例：

```text
张哥：
- 爱打杭麻财敲，0.5 档
- 也打川麻，1-32 档，偏好换三张
- 通常一个人来，置信度 0.9
- 常在晚上 19:00-20:00 活跃
```

这类画像会影响追问策略。比如老客户张哥只说“今天下班有人打麻将吗”，系统可以先按“张哥一个人，等价于 `173`”来记，只追问时间和档位；新客户或低置信客户仍要先确认“你这边几个人”。

### 6. 候选人推荐

当一桌局可以开始找人时，系统会生成候选人排序。

评分因素：

- 玩法是否匹配。
- 档位是否匹配。
- 是否偏好无烟。
- 是否常在这个时间段打。
- 是否当天已经打过。
- 是否距离上一场太近。
- 是否刚被邀请过。
- 是否已在另一桌有效局里。
- 是否最近拒绝较多。

推荐结果包含：

- 客户名。
- 分数。
- 推荐原因。
- 风险提示。

### 7. 防止重复拉人

这是麻将馆真实运营里很重要的能力。

系统保证：

- 同一个客户不会被同时安排进多个有效局。
- 客户接受一个邀约后，其他待确认邀约自动废弃。
- 某桌已满后，后续报名会收到候补或已满提示。
- 局结束、取消、过期后客户锁释放。

### 8. 多局隔离

每一桌局都有独立 `game_id`。

系统不会把不同局的消息、邀约、报名混在一起：

- 群 A 的 19:00 财敲局。
- 群 B 的 19:30 川麻局。
- 私聊里的候选人确认。

这些状态都独立追踪。

### 9. 生命周期管理

局状态包括：

- 待补充信息
- 开放找人
- 邀约中
- 已占位
- 已组好
- 已完成
- 已取消
- 已过期

系统会自动推进生命周期：

- 已确认的局按预计时长结束。
- 没说时长时默认 4 小时。
- 结束后留 30 分钟缓冲。
- 未组满的局过开局时间 30 分钟后过期。

### 10. 群发和私聊草稿

系统生成两类草稿：

- 群发组局草稿。
- 私聊邀约草稿。

示例：

```text
组局，杭麻，财敲，0.5档，06-16 19:30，缺1位，无烟。方便的朋友私我。
```

```text
张哥，今晚 19:30 有一桌，杭麻，财敲，0.5档，还差1位，无烟，你方便来吗？
```

当前策略是先进入 outbox，默认人工确认后再发送。

### 11. LLM 语义兜底

规则解析是主路径，LLM 是兜底。

LLM 调用前会先经过 `ContextBuilder`：

- 只注入当前任务需要的消息、画像、局状态、房态和玩法词典。
- 手机号、微信号、长数字和资金相关信息会脱敏。
- 使用稳定引用替代原始用户 ID、局 ID 和房间 ID。
- 记录 `context_digest` 和脱敏上下文快照，便于审计和事故复盘。
- 按预算裁剪旧消息和旧局，避免长上下文噪声。

LLM 适合处理：

- 新行话。
- 模糊口语。
- 上下文依赖表达。
- 语音转写不完整。
- 图片 OCR 噪声。

LLM 不直接改状态，只输出结构化解释或规范化文本，再交给核心状态机处理。

这样既能利用模型理解能力，又避免模型直接做不可控业务动作。

### 12. 安全和人工兜底

系统遇到以下内容会转人工：

- 抽水
- 赌资
- 赌博
- 洗钱
- 上分/下分
- 代收/代付
- 结算输赢
- 放贷
- 借码

异常、超时、LLM 返回不可解析结果，也会 fail-closed 转人工。

## 生产化能力

当前核心已经具备：

- 幂等处理。
- 同会话保序。
- 审计日志。
- LLM 上下文 digest 和脱敏快照。
- outbox 草稿。
- 状态快照。
- 运行指标。
- 异常和超时保护。
- LLM 开关和模型配置。

后续上线需要替换或补齐：

- 微信/企业微信 adapter。
- 生产数据库。
- 分布式队列。
- 分布式客户锁。
- 管理后台。
- 多租户权限。
- 真实发送通道。

## 商业价值

对门店：

- 减少老板和店员反复看群、拉人、追问、记偏好的时间。
- 提升缺人局的补位速度。
- 降低重复打扰客户的概率。
- 增强客户画像沉淀。
- 提升私域群的运营效率。

对 SaaS 产品：

- 可抽象为“本地玩法词典 + 私域客户画像 + 组局调度 Agent”。
- 可以服务不同城市、不同玩法、不同门店规模。
- 可逐步扩展到棋牌室排班、会员营销、活动召回、短视频获客等模块。

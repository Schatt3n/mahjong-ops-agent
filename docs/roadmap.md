# 产品路线图

本文档描述从当前核心引擎到可商业化上线产品的演进路径。

## 当前阶段：Core Engine

当前仓库已经完成的是核心引擎。

目标：

- 证明麻将馆运营问题可以被结构化。
- 建立可测试的业务状态机。
- 建立可靠消息处理骨架。
- 预留 LLM 语义兜底。

已具备：

- 规则解析。
- 本地玩法体系。
- 客户画像。
- 候选人推荐。
- 客户锁。
- 局生命周期。
- outbox 草稿。
- 幂等和保序。
- 审计和指标。
- LLM 接入点。

## 阶段 1：真实消息接入和人工审核

目标：

让真实门店可以把消息接进系统，但所有对外发送仍由人工确认。

需要实现：

- 微信/企业微信 adapter。
- 真实消息标准化。
- 图片 OCR 管线。
- 语音 ASR 管线。
- outbox 审核台。
- 基础客户画像管理。
- 基础玩法词典管理。
- 操作审计。

上线策略：

- 不自动私聊客户。
- 不自动处理敏感内容。
- 所有 LLM 参与过的结果默认可查看解释。

衡量指标：

- 组局识别准确率。
- 人工修改草稿比例。
- 私聊邀约接受率。
- 转人工率。
- 误判率。

## 阶段 2：半自动组局

目标：

对高置信、低风险场景自动化，减少店员重复操作。

需要实现：

- 高置信群发自动发送。
- 常见玩法自动建局。
- 常见报名自动占位。
- 客户冲突自动拒绝。
- 候补机制。
- 改时间协商流程。
- 待组局队列运营视图。
- LLM 工具调用审计。

上线策略：

- 私聊邀约可按客户分层开放自动发送。
- 新客户、敏感客户、高价值客户仍需人工确认。
- LLM 低置信结果继续转人工。

衡量指标：

- 自动处理率。
- 组局成功率。
- 客户打扰投诉率。
- 平均成局时间。
- 店员节省时间。

## 阶段 3：客户运营和营销

目标：

从“帮老板拉人”升级为“客户运营系统”。

需要实现：

- 客户分层。
- 沉睡客户召回。
- 玩法偏好分群。
- 到店频率分析。
- 活动触达。
- 生日/节假日关怀。
- 抖音/小红书素材建议。
- 营销效果归因。

上线策略：

- 自动营销必须有频率限制。
- 所有营销消息进入 outbox。
- 客户可设置 no-contact。

衡量指标：

- 复购率。
- 召回率。
- 客户活跃度。
- 活动转化率。
- 客单价或桌台利用率。

## V4：多店 SaaS

目标：

支持多家门店、多租户、权限、计费和运营后台。

需要实现：

- 多租户数据隔离。
- 门店管理。
- 员工账号和权限。
- 套餐计费。
- API key 管理。
- 模型成本统计。
- 系统健康监控。
- 门店级玩法词典。
- 门店级客户画像。
- 多店数据看板。

衡量指标：

- 单店月活。
- 单店日均消息量。
- 自动处理率。
- LLM 成本/店/月。
- 客户留存。
- 付费转化。

## V5：行业 Workflow 平台

目标：

把麻将馆运营 workflow 抽象成可复用的“私域运营 agentic workflow 平台”。

可拓展方向：

- 棋牌室。
- 台球厅。
- 剧本杀。
- 私教工作室。
- 本地生活预约型门店。

共性能力：

- 私域消息理解。
- 客户画像。
- 拼局/预约/排班。
- 候选人推荐。
- 营销召回。
- outbox 审核。
- 多租户 SaaS。

## 技术债和风险

需要尽早处理：

- 玩法词典不能长期写死在代码里。
- SQLite 只能用于本地和早期验证。
- LLM 成本必须可观测。
- 微信通道合规和稳定性需要单独评估。
- 自动发送策略必须谨慎。
- 客户隐私和数据权限需要产品化设计。
- 真实语音、图片质量会显著影响理解效果。

## 推荐下一步

1. 把玩法词典抽成配置表。
2. 增加客户画像管理 API。
3. 增加 outbox 审核台。
4. 设计生产数据库 schema。
5. 接入一个真实但低风险的消息来源。
6. 收集 500-1000 条真实聊天样本做评测集。
7. 增加 LLM 成本、调用日志、失败率统计。
8. 做第一版门店运营看板。

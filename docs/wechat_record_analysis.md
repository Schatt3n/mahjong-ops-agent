## 真实语言现象

1. 财敲/杭麻是主干玩法

真实群里大量出现 `cq371`、`cq272`、`cq173`、`财敲`、`杭麻`。这里 `cq` 在当前业务语境中是杭麻“财敲”，不是重庆麻将。代码中已经按本地业务含义处理。

2. 群公告经常是字段模板，不是自然语言

高频模板包括：

- `人数：272`
- `时间：人齐开`
- `大小：0.5`
- `🚬：无烟`
- `256封顶`、`256上限`

所以 agent 不能只做自然语言理解，也要支持半结构化字段解析。

3. 烟况比“有烟/无烟”更细

真实数据中出现大量 `少烟`、`烟都可`、`都行`、`🚬：无/有/少/都可`。客户画像里需要记录“强无烟”“可少烟”“烟都可”“可吸烟”，否则推荐候选人时会粗糙。

4. 一个消息里经常合并多桌

真实群里经常一条消息发多桌，例如：

- 第 1 桌：`8点，1无烟，371`
- 第 2 桌：`9点，1有烟，173`
- 另起一行：`红中368，272`

当前核心模型仍是一条消息生成一个主 `GameRequest`，这会造成“只吃到其中一桌”或“多桌信息串成噪声档位”。这是下一阶段最应该补的能力。

5. 红中玩法也有自己的数字语法

真实数据里有：

- `红中368`
- `红中368/568`
- `红中568/3块爆炸`
- `红中368鲨鱼`

这些不是普通“底注-封顶”语义，应该作为红中特定档位/玩法记录。

6. “订房/房间/满房”是独立业务线

样本里有 `订房`、`房间`、`满房`、`只剩几点后的房间`、`帮我约个四小时房间`。这不是纯组局意图，而是“房态/订房/组局”三者之间的协同问题。当前已加入基础房态容量和满房时间协商，后续还应继续完善单独的 `room_booking` / `room_availability` intent。

## 剩余关键缺口

1. 多桌消息拆分

这是最大缺口。真实消息经常一条包含多桌，当前一条消息只生成一个主局，会让局之间互相污染。

建议新增 `MultiGameExtractor`：

- 先按编号、换行、玩法前缀、时间字段切分候选片段。
- 每个片段独立调用 `MahjongMessageParser`。
- 输出 `list[GameRequest]`，并保留同一源消息的 `source_message_id`。
- 对疑似切分失败的片段进入人工复核队列。

2. 订房/房态 intent

应从组局主链路中拆出：

- `room_booking_request`
- `room_availability_update`
- `room_time_conflict`
- `room_duration_request`

这样 agent 才能处理“今晚满房，只剩 9 点后”“帮我订 4 小时房间”“这桌人齐但没有房间”等真实运营问题。

3. 客户画像自动沉淀

应该从真实聊天和成局结果中持续更新画像：

- 常打玩法：杭麻、财敲、红中、川麻、捉鸡
- 常打档位：0.5、1、1-32、368
- 烟况偏好：无烟、少烟、烟都可、可吸烟
- 时间习惯：下班后、通宵、周末、下午场
- 常见同行人数：一个人来、两个人来、经常带朋友
- 社交约束：女生局、情侣可否、熟人局
- 疲劳度参数：一天多场能力、两局间隔、邀约冷却

画像写入应带证据、置信度和时间衰减，避免一次聊天误伤长期偏好。

4. 地方玩法词库和未知词队列

地方麻将的缩写非常强业务相关。建议维护一份可热更新的 `MahjongOntology`：

- `cq = 杭麻财敲`
- `371 = 三缺一`
- `272 = 二缺二`
- `173 = 一缺三`
- `216 = 2-16`
- `368/568`、`568/3` 等红中本地玩法

遇到未知高频词，不要立即乱猜，应进入“未知词队列”，由老板确认后加入词库。

5. LLM 辅助理解要接在规则之后

真实上线时，LLM 应用于：

- 模糊自然语言意图判断
- 图片 OCR/语音 ASR 后的语义归一
- 未知玩法词解释
- 多桌拆分失败后的低置信度补救
- 给人工运营输出解释和建议

LLM 不应直接改状态，应该输出结构化候选事实，再由规则校验、状态机和审计日志接管。

## 产品结论

这份真实聊天记录说明，麻将馆运营 agent 不是“聊天机器人”，而是一个本地业务语义引擎：

- 前层要能吃微信里非常脏、非常短、非常本地化的消息。
- 中层要有可解释的状态机，保证局不串、人不重复、消息有序。
- 后层要有客户画像、邀约疲劳度、房态、审计和人工兜底。
- LLM 是增强理解能力，不是替代业务规则和状态一致性。

当前项目已经具备核心组局状态机、客户锁定、疲劳度、基础房态冲突处理、LLM 接入点、可观测日志和真实语料驱动的解析增强。下一步最值得投入的是“多桌消息拆分”和“完整订房/房态业务线”，这两项会显著提升真实上线可用性。

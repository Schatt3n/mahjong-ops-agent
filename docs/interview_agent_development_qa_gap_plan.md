# Agentic Workflow 开发面试问答与差距计划

本文档用于面试时讲清楚 Mahjong Ops Workflow 的技术设计、当前实现、未完成能力和生产化计划。

状态说明：

- 已做到：当前代码已经有对应实现和测试。
- 部分做到：已有核心骨架，但还不够生产化。
- 未做到：当前主要停留在设计文档或待实现阶段。

## 一句话项目定位

这是一个面向麻将馆/棋牌室私域运营的 agentic workflow。它不是简单聊天机器人，也不是完全自治 Agent，而是一个围绕“识别组局意图、维护客户画像、推荐候选人、避免重复拉人、生成可审核 outbox”的可控运营决策系统。

当前设计原则：

1. 规则和状态机是主路径。
2. LLM 只做语义兜底和归一化，不能直接改状态或发消息。
3. 所有对外发送先进入 outbox。
4. 消息处理必须幂等、保序、可回溯。
5. 敏感资金、结算、抽水等场景 fail-closed 转人工。

## 1. 产品与边界

| 问题 | 推荐回答 | 当前是否做到 | 补齐计划 |
| --- | --- | --- | --- |
| 这个 Agent 替代老板哪些工作？哪些必须转人工？ | 替代低风险、重复性运营工作：识别组局、追问缺失信息、生成群发/私聊草稿、推荐候选人、维护画像和局状态。必须转人工的包括资金/结算/抽水、低置信语义、客户投诉、高价值客户特殊处理、真实发送前审核。 | 部分做到。识别、追问、建局、推荐、outbox、敏感转人工已实现；真实人工审核台未实现。 | 做 outbox 审核台；增加风险等级；将“自动发送策略”按租户/客户分层配置。 |
| “用户想打麻将”的意图边界怎么定义？ | 通过多模态证据判断：文本关键词、玩法/档位/几缺几、时间、找人信号、语音转写、OCR、表情描述。明确组局优先进入状态机，弱意图追问，无关群聊静默。 | 已做到核心。`signals.py`、`parser.py`、`responder.py` 已支持弱意图、语音/OCR/表情 metadata。 | 增加真实 ASR/OCR adapter；沉淀真实样本评估集；支持图片多桌切分。 |
| 如何区分潜在客户咨询、明确组局、加入已有局、取消/满员/变更规则？ | 用意图分类加状态上下文：弱表达为 potential lead，完整玩法/时间/人数为 create/queue invites，报名词命中 active invitation 为 accept seat，满员/取消/拒绝进入对应状态。 | 部分做到。潜在咨询、明确组局、报名、取消/满员/拒绝已有逻辑；复杂变更规则不足。 | 增加 `update_game` 状态迁移：改时间、改玩法、改人数、改无烟/有烟，并保留审计。 |
| 最小可上线版本是什么？ | V1 不是全自动老板，而是“真实消息接入 + 状态识别 + 草稿审核 + 人工发送”。上线目标是减少老板整理信息和写邀约的时间，不直接承诺全自动经营。 | 部分做到。核心引擎与控制台/HTTP 可用；真实消息接入和审核台缺失。 | V1 里先接企业微信/低风险消息源；所有发送进入审核台；上线指标看误判率和人工修改率。 |
| 商业化前必须补齐什么？ | 真实通道、审核台、玩法词典配置、画像管理、生产数据库、权限/多租户、监控告警、隐私合规、评估集。 | 部分做到。架构文档已有路线图，核心能力已有。 | 以 V1 到 V4 规划推进：真实 adapter、后台、PostgreSQL、多租户、监控计费。 |

## 2. Agent 架构

| 问题 | 推荐回答 | 当前是否做到 | 补齐计划 |
| --- | --- | --- | --- |
| 单体 Agent 还是多 Agent？为什么？ | 当前不是多 Agent 系统，而是单体 agentic workflow：一个 responder 编排规则、ContextBuilder、LLM、状态机。原因是组局业务强一致，过早多 Agent 会增加状态冲突。未来可拆成语义、调度、营销等内部模块，但写状态仍由核心状态机统一执行。 | 已做到单体 workflow。 | 后续按职责拆内部模块，不让多个智能模块直接写同一状态。 |
| 规则解析、LLM、状态机、工具分别负责什么？ | 规则解析负责确定性抽取；LLM 负责低置信语义归一；状态机负责建局、邀约、锁、生命周期；工具负责外部查询或草稿生成，写操作必须回到状态机。 | 部分做到。规则、LLM、状态机已实现；工具注册中心未实现。 | 增加 `ToolRegistry`，先只读工具：玩法词典、客户画像、当前待组局、历史摘要。 |
| 为什么不能让 LLM 直接改数据库、直接发消息？ | 因为 LLM 输出不稳定，可能幻觉或误解。生产系统要求幂等、审计、权限、回滚和一致性，所以 LLM 只能输出结构化事实，状态变更由确定性代码执行。 | 已做到。`llm.py` 只返回 `LLMResolution`，`responder.py` 再交给 parser/core。 | 保持边界；工具调用也采用同样原则。 |
| LLM 是主决策器还是兜底语义解析器？ | 兜底语义解析器。规则优先更稳定、便宜、可测试；LLM 处理新行话、模糊话、上下文依赖。限制是复杂自然语言可能需要更强的上下文和工具查询。 | 已做到。已接 Qwen/DashScope，规则失败或疑似相关时调用。 | 增加 LLM 调用成本、失败率、低置信转人工统计。 |
| 多平台输入输出如何抽象？ | 输入都转换为统一 `IncomingEnvelope/Message`，输出都写入 `outbox_events`，再由 `OutputRouter` 发到指定通道。核心引擎不关心微信、小红书、抖音差异。 | 部分做到。控制台/API/微信测试输出已有；小红书/抖音真实 adapter 未实现。 | 定义 Channel Adapter SDK；实现 WeChat/企微/XHS/Douyin 的 inbound/outbound adapter。 |

## 3. 状态机

| 问题 | 推荐回答 | 当前是否做到 | 补齐计划 |
| --- | --- | --- | --- |
| 一桌局从创建到结束有哪些状态？ | `need_clarification`、`open`、`negotiating`、`holding`、`confirmed`、`completed`、`cancelled`、`expired`。 | 已做到模型层；核心常用状态已覆盖。 | 补全 `negotiating/holding` 的真实业务流。 |
| 三缺一、二缺二、一缺三、待确认、已成局、取消、过期怎么建模？ | 用 `current_player_count` 和 `missing_count` 表达几缺几，用 `GameStatus` 表达生命周期，用 invitation 状态表达客户级确认。 | 已做到核心。 | 增加候补队列和多人确认流程。 |
| 用户早上打一局，晚上又来组局怎么判断？ | 用客户画像里的 `max_games_per_day`、`min_hours_between_games`、疲劳度和已完成局记录判断。高频玩家可降低限制，普通玩家默认一天一次。 | 已做到测试覆盖。 | 后续把疲劳度策略做成门店配置，并记录真实到店/离店时间。 |
| 用户已锁到一个局，又响应另一个局怎么处理？ | 客户锁优先级高于 LLM 和文本意图。已在有效局时拒绝重复安排，或提示已帮他占位。 | 已做到。 | 增加跨平台同人识别，解决微信号/昵称/手机号映射。 |
| 牌局规则变更如何迁移状态？ | 理想做法是产生 `game_update` 事件，更新字段、重算候选人、废弃不再匹配的邀约、通知人工确认。 | 部分做到。当前可以建局和处理部分取消/满员，复杂变更不足。 | 实现 `GameUpdateCommand` 和事件日志；支持改时间、改档位、改玩法、改人数。 |

## 4. 上下文管理

| 问题 | 推荐回答 | 当前是否做到 | 补齐计划 |
| --- | --- | --- | --- |
| 如何保证不同用户/群/平台上下文不串？ | 用 `tenant_id + channel_id` 形成 `conversation_id`，每个消息有 `source_message_id` 和 sequence，同会话保序，不同会话隔离。 | 已做到本地版本。 | 生产改为 PostgreSQL/队列分区；增加跨平台用户身份映射表。 |
| 私聊和群聊上下文怎么合并？ | 不直接混在一个 conversation，而是用 customer identity 关联：群聊局状态、私聊确认和客户画像通过 `customer_id` 连接。 | 部分做到。客户画像和邀约可连接，但真实跨通道身份合并未做。 | 实现 `identity_links`：wxid、手机号、昵称、平台 user id 到 customer id。 |
| 用户分多句才表达完整意图怎么办？ | 同一用户、同一会话、短时间窗口内的碎片消息会先合并理解，例如“老板 / 今天下午 / 有没有打麻将的 / 0.5或者1都行 / 烟也都可”。系统会按“今天下午、0.5或1、烟况不限”追问缺失信息；如果是高置信老客且画像显示通常一个人来，则按 `173` 补全人数，不再重复问“几个人”。 | 已做到核心。`AgentResponder` 有短窗口碎片聚合，`AgentCore` 会用 `usual_party_size` 补全人数，并有回归测试。 | 生产 adapter 层再加 2-5 秒 debounce，等用户连发结束后减少重复追问。 |
| 用户说“还是上次那个”怎么找上次含义？ | 从客户画像、历史已完成局、最近会话摘要中取玩法/档位/规则，再由 LLM 或规则确认。不能确定时追问。 | 部分做到。LLM 能理解“上次那个”，但没有长期历史检索。 | 增加历史摘要表和最近偏好检索工具；将上次玩法写回画像。 |
| 上下文存多久？哪些进长期画像？ | 短期会话保留组局上下文和最近 N 轮；长期画像只存稳定偏好、频率、拒绝/接受统计、风险标签，不存无关聊天。 | 部分做到。运行时有短期上下文，画像模型存在；长期归档策略未产品化。 | 做数据保留策略和画像更新任务；增加用户隐私/删除能力。 |
| LLM 使用历史消息如何避免太长和隐私泄露？ | 不把全量聊天直接塞给 LLM，只提供结构化摘要和必要事实；敏感字段脱敏；高风险租户可关闭 LLM。 | 已做到核心。`ContextBuilder` 已支持上下文裁剪、脱敏、稳定引用、预算控制、`context_digest` 和脱敏快照。 | 生产继续补租户级 LLM 开关、长期历史摘要检索、隐私删除能力和 token 成本统计。 |

## 5. 并发与可靠性

| 问题 | 推荐回答 | 当前是否做到 | 补齐计划 |
| --- | --- | --- | --- |
| 多用户同时组局如何保证不乱序？ | 同一 `conversation_id` 通过 sequence 串行处理，不同 conversation 可并行。状态写入使用事务，客户锁避免同人重复占位。 | 已做到本地 SQLite 版本。 | 生产用队列按 conversation 分区，数据库事务和唯一约束保证一致性。 |
| 同一用户同时被多个局邀请如何防重复占位？ | 邀请前检查 active invitation 和 customer lock，接受一个邀约后废弃其他待确认邀约。 | 已做到。 | 增加分布式锁或数据库唯一索引：`active_customer_lock(tenant_id, customer_id)`。 |
| 分布式下节点挂了，未处理消息如何恢复？ | 消息先入队和持久库，worker claim lease；节点挂了 lease 到期后别的 worker 继续。outbox 独立派发，发送结果幂等记录。 | 部分做到。本地 SQLite 能重启恢复，未实现生产 lease/队列。 | 引入 Kafka/Redis Stream + PostgreSQL；实现 message lease、dead letter、retry backoff。 |
| outbox 为什么持久化，不能直接发？ | 直接发会遇到进程崩溃、重复发送、无审计、无法人工审核。outbox 能保证先记录意图，再可靠派发和回溯。 | 已做到。 | 增加审核台、重试策略、发送回执对账。 |
| 幂等键怎么设计？重复微信消息怎么办？ | inbound 用 `tenant_id + source_message_id` 去重；用户真的重复发送时，用同会话、同发送人、同消息类型、规范化文本和短时间窗口生成语义指纹；outbox 用业务维度生成 idempotency key，如 `game_id + customer_id + private_invite`。 | 已做到本地版本。 | 真实 adapter 必须保留平台原始 msg id；生产语义去重还应加入 active game context、平台时间戳和 gap timeout。 |

## 6. LLM 接入

| 问题 | 推荐回答 | 当前是否做到 | 补齐计划 |
| --- | --- | --- | --- |
| 什么时候调用 LLM？ | 规则解析失败但疑似麻将相关、私聊/手动入口、出现新行话或上下文依赖时调用。敏感词先转人工，不调用 LLM。 | 已做到。 | 增加可配置 LLM 策略：按租户、渠道、置信度、成本预算控制。 |
| LLM 超时、JSON 错、返回不稳定怎么办？ | fail-closed：转人工或静默；保留错误日志和 trace；不让 LLM 结果直接写状态。 | 已做到基础。 | 增加重试、模型降级、输出 schema 校验更严格化。 |
| 如何防止 `cq` 被幻觉成重庆麻将？ | 本地规则优先，prompt 明确 `cq=杭麻财敲`，LLM 输出 normalized_text 后仍由规则解析校验。 | 已做到并实测。 | 把玩法词典从 prompt/代码抽成门店配置和可检索工具。 |
| 本地行话如何沉淀？ | 高频稳定行话进规则/词典；模糊表达进评估集和 prompt；大量样本后再考虑微调；低频新词用 LLM/RAG。 | 部分做到。已有规则和 prompt，真实样本分析文档有。 | 做玩法词典管理后台；建立错例闭环；收集 500-1000 条评测集。 |
| 模型成本高怎么办？ | 规则优先、缓存 LLM 结果、低成本模型初筛、高成本模型只处理难例、批量离线总结、监控 token 成本。 | 部分做到。规则优先已做；成本统计/缓存未做。 | 实现 LLM usage logging、缓存、模型路由。 |

## 8. 评估与训练

| 怎么证明 Agent 回复是对的？ | 用场景评估集、单测、回放真实消息、人工标注、关键指标监控。判断不只是回复文本，还包括状态、outbox、锁和审计。 | 部分做到。有单测和 scenario eval。 | 增加真实聊天回放评估和人工标注集。 |

## 生产化补齐计划

### P0：让项目能安全试点

目标：真实门店可以接入消息，但不自动发私聊。

1. 实现 outbox 审核台。
2. 接入一个真实 inbound adapter，优先企业微信/客服 API 或人工导入。
3. 抽象玩法词典配置，不再长期写死代码。
4. 实现客户画像管理 API。
5. 增加 trace 回放脚本。
6. 建立 200 条真实样本评估集。

验收标准：

- 任意消息能在 UI 看到 trace、解析结果、状态变化、outbox 草稿。
- 人工确认后才发送。
- 真实样本评估通过率可量化。

### P1：半自动组局

目标：高置信、低风险场景自动处理，降低老板操作量。

1. 支持 `update_game`：改时间、改规则、改人数。
2. 增加候补机制。
3. 增加跨平台身份映射。
4. 做客户疲劳度和 no-contact 配置。
5. 增加 LLM 成本、失败率、低置信率监控。
6. 增加发送回执和 outbox retry/backoff。

验收标准：

- 高置信清晰组局自动生成草稿和候选人。
- 客户不会被重复拉入两个有效局。
- 发送失败可追踪、可重试、不会重复骚扰。

### P2：生产级可靠性

目标：支持多 worker、节点重启、消息堆积和可观测。

1. SQLite 迁移到 PostgreSQL。
2. 引入 Kafka/Redis Stream/RabbitMQ。
3. 实现 message lease、dead letter、retry。
4. 引入 OpenTelemetry/Prometheus。
5. 实现租户、员工权限、审计报表。

验收标准：

- 单节点挂掉后未处理消息可由其他节点继续。
- 同会话消息不乱序。
- 所有发送动作可审计和回放。

### P3：商业化 SaaS

目标：支持多门店、多租户、配置化和计费。

1. 多租户数据隔离。
2. 门店级玩法词典和话术模板。
3. 门店级客户画像和分群。
4. API key/模型成本/套餐管理。
5. 运营看板：识别率、成局率、转人工率、投诉率、LLM 成本。

验收标准：

- 新门店可以通过后台配置玩法和客户池。
- 单店成本和自动化效果可量化。
- 敏感数据有权限控制和删除能力。

## 面试表达建议

可以这样总结：

> 我没有把这个项目设计成“LLM 接微信然后自由聊天”的机器人，也不把当前版本包装成完全自治 Agent。它更准确是一个有状态、有审计、有 outbox、有客户锁的 agentic workflow。LLM 在这里是语义增强层，不是业务状态机。真正难的不是让模型回复一句话，而是在多用户、多局、多平台、乱序和重复消息下，保证不串局、不重复拉人、可回溯、可人工接管。

这句话能把项目和普通 ChatBot 区分开，也能体现 Agentic Workflow 的工程化能力。
